## 1. Product Goal Check

DynEcho is intended to behave like a professional acoustic calculator, not a marketing shell.

Target outputs include:

- `Rw`, `R'w`
- `Dn,w`, `Dn,A`, `DnT,w`, `DnT,A`, `DnT,A,k`
- `Ln,w`, `Ln,w+CI`, `L'n,w`, `L'nT,w`, `LnT,A`
- `DeltaLw`

Success means:

- numbers stay inside defended mathematical corridors
- evidence posture is explicit
- unsupported lanes remain explicit instead of fabricated
- UI/UX supports a professional workflow without hiding solver posture

## 2. Current Reality

### 2.1 Engine

The engine is now materially beyond the earlier migration shell.

Live today:

- calibrated airborne outputs: `Rw`, `STC`, `C`, `Ctr`
- field airborne continuation: `R'w`, `Dn,w`, `Dn,A`, `DnT,w`, `DnT,A`, `DnT,A,k`
- exact impact import lanes
- exact floor-family impact rows
- official impact product rows
- heavy-floor scoped predictor lane
- labeled low-confidence published-family fallback lanes across the remaining floor corridors

Recent hardening that is now in place:

- impact-only upstream parity is green
- assembly impact parity is green
- low-confidence floor-family fallback behavior is aligned with upstream instead of drifting locally
- timber bare-floor low-confidence lanes expose finite airborne companions when they genuinely exist
- field-airborne output posture is now described through one shared truth set instead of scattered UI rules

Current explicit non-goals / unsupported lanes:

- `IIC`
- `AIIC`
- `NISR`
- `ISR`
- `LIIC`
- `LIR`
- `HIIC`

Those outputs remain intentionally staged.

### 2.2 Web Product

The web product now has two real surfaces:

- guided shell: primary product candidate
- advanced desk: expert/operator surface

The guided shell is currently the stronger product face because it explains corridor posture with less operator overhead.

Recent web hardening that is now in place:

- `/workbench` standalone start path no longer returns `500`
- chart panels are guarded against zero-size startup rendering
- `DnT,w` / `DnT,A` no longer falsely block on `RT60`
- the real guided blocker for room-standardized field outputs is now presented as partition geometry plus receiving-room volume
- field-airborne provenance is now visible in live result cards, delivery assist, and report/export surfaces
- consultant-facing decision trail is now shared between the report UI and markdown export instead of being scattered across separate status notes

### 2.3 Delivery Posture

The app is no longer only a calculation shell. It now has working product-level surfaces for:

- requested-output tracking
- capability mapping
- field-risk framing
- scenario comparison
- report/export composition
- field-airborne provenance framing

This is better, but still not full consultant-grade delivery. The product now exposes more truth, but the reporting layer still needs a stronger audit trail and clearer memo-grade narrative.

## 3. What Is Still Missing

### 3.1 Math / Solver Gaps

The biggest remaining gaps are scope gaps, not random defects:

- ASTM impact families remain staged
- field continuation needs deeper benchmark coverage and stronger parity discipline
- some guide-driven outputs still rely on honest bounded posture rather than full standards-backed solver depth

### 3.2 Product Gaps

The product still needs:

- stronger consultant-facing report narrative
- clearer assumptions and source citation surfaces
- a more explicit audit trail for why a lane was exact, derived, companion, or low-confidence
- less density in the advanced desk
- a more deliberate primary workflow around the guided shell

## 4. Recommended Execution Order

### Phase 1 — Consultant-Grade Reporting

Priority:

- source references
- confidence posture
- active corridor
- assumptions
- memo-grade export narrative
- decision/audit trail

Exit criteria:

- consultant can lift an output into a brief or memo without re-explaining the solver posture by hand

### Phase 2 — Guided Shell As Primary Product

Priority:

- treat the guided shell as the default professional workflow
- keep advanced desk as expert/operator mode
- reduce lane confusion and panel density

Exit criteria:

- first-time professional user can understand output posture without opening diagnostics
- advanced desk becomes optional depth, not required interpretation

### Phase 3 — Field Output Hardening

Priority:

- `R'w`
- `Dn,w`
- `Dn,A`
- `DnT,w`
- `DnT,A`
- `DnT,A,k`

Exit criteria:

- benchmark-backed corridors
- explicit parity acceptance where upstream comparison is meaningful
- clean target-output posture on both engine and UI surfaces

### Phase 4 — ASTM / Extended Impact Families

Priority:

- `IIC`
- `AIIC`
- `NISR`
- `ISR`
- `LIIC`
- `LIR`
- `HIIC`

Exit criteria:

- standards-backed adapters, not placeholder marketing outputs

## 5. Product Judgement

DynEcho is now clearly beyond the "migration demo" stage.

It is not yet a universal consultant-grade acoustic calculator, but it is already a serious web product foundation with:

- a defended engine boundary
- green parity on the repaired impact lanes
- explicit evidence posture
- a credible guided workflow
- visible field-airborne provenance in both UI and export surfaces

The next highest-value move is now consultant-grade reporting and audit posture, followed by guided-shell productization and then deeper field-output hardening.
