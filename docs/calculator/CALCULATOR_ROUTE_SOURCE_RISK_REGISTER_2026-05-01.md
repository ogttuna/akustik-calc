## R1 - Flat-List Route-Family Flip

Symptom:

Small layer swaps can move the route family between triple-leaf,
double-leaf, lined-massive, masonry/nonhomogeneous, or multi-leaf
screening lanes. The rockwool user defect is the concrete example:
moving one `50 mm` rockwool layer away from the other changes the live
answer from the double-leaf control to a low-confidence multi-leaf
screening result with roughly a 10 dB difference.

Why it can happen:

The flat layer list does not say which layers form `sideALeaf`,
`cavity1`, `internalLeaf`, `cavity2`, or `sideBLeaf`. The route
classifier therefore infers topology from adjacency, material class,
and thickness. That inference is useful for screening but is not enough
for exact triple-leaf behavior.

Current guard evidence:

- `dynamic-airborne-order-sensitivity.test.ts`
- `dynamic-airborne-instability-repro.test.ts`
- `dynamic-route-instability.test.ts`
- Gate I grouped topology input workbench tests
- Gate J company-internal acceptance rehearsal for adjacent and
  split-rockwool user stacks

Remaining gap:

Grouped topology can now be represented, but exact runtime remains
blocked because local rockwool, local gypsum board, MLV, plaster, local
support topology, and source tolerance are not owned together. Sibling
wall families can still hit route-boundary holds when the flat list is
ambiguous.

Fix strategy:

- Keep exact promotion blocked from flat-list inference alone.
- Require grouped topology for triple-leaf exact candidates:
  `sideALeaf`, `cavity1`, `internalLeaf`, `cavity2`, `sideBLeaf`,
  `internalLeafCoupling`, and `supportTopology`.
- Keep route-card diagnostics explicit when the route is inferred,
  ambiguous, or source-validation blocked.
- Only move runtime after source rows or calibrated curves, local
  material mapping, tolerance owner, negative boundaries, and paired
  visible tests pass.

Tests to add or keep:

- Keep the adjacent/split rockwool paired value tests.
- Add a route-family flip matrix for every new exact wall lane before
  runtime promotion.
- Add web route-card assertions that distinguish missing topology from
  complete-but-source-blocked topology.

## R2 - Duplicate / Many-Layer Stack Drift

Symptom:

Duplicating layers or pasting a large repeated stack can shift the
best-fit family or support status even when all numeric outputs remain
finite. A user can unintentionally duplicate a board/fill package or
paste a wall twice and receive a plausible-looking but unowned answer.

Why it can happen:

Heuristic family scoring reacts to leaf count, mass distribution,
cavities, and porous fills. A duplicated stack may look like a new
multi-leaf construction rather than a bad edit. A finite value only
proves numerical stability; it does not prove acoustic correctness.

Current guard evidence:

- `raw-wall-hostile-input-answer-matrix.test.ts`
- `floor-many-layer-stress-gate-a-matrix.test.ts`
- `apps/web/features/workbench/floor-many-layer-stress-gate-a-card-matrix.test.ts`
- Gate J many-layer / reorder hostile input acceptance coverage
- Workbench duplicate-role floor warnings in `workbench-store.test.ts`

Remaining gap:

Duplicate-stack semantics are not source-owned for wall exact lanes.
The calculator can remain finite and renderable while still lacking a
defensible physical interpretation.

Fix strategy:

- Treat duplicate-stack exact promotion as blocked unless the user has
  explicit grouped topology or a known source-row construction that
  allows repeated packages.
- Add a route diagnostic for suspicious duplicate packages when family
  selection changes.
- Preserve low-confidence screening or family-boundary hold when
  duplicate edits create unclear topology.
- Do not coalesce or normalize duplicate layers globally when position
  changes the physics.

Tests to add or keep:

- Add duplicate-stack route-card and value tests for every new wall
  exact lane.
- Keep many-layer stress tests focused on finite output plus visible
  warning honesty, not only lack of crashes.

## R3 - Masonry / Lined-Massive Boundary Drift

Symptom:

AAC, pumice, concrete, silicate, board/fill/gap, and hybrid wall
stacks can sit near the boundary between `masonry_nonhomogeneous`,
`lined_massive_wall`, and `double_leaf`. A small board, cavity, or fill
change can move the family or trigger a family-boundary hold.

Why it can happen:

Massive-core and lined-massive routes depend on substrate mass,
lining-cavity coupling, board position, and whether the added layer is
an attached finish or an independent leaf. Flat lists often omit the
coupling and substrate identity needed to decide that boundary.

Current guard evidence:

- `dynamic-airborne-family-boundary.test.ts`
- `dynamic-airborne-family-boundary-scan.test.ts`
- lined-masonry Knauf `MWI.2A` mapping/tolerance no-runtime gates

  masonry, and one-side lining cases

Remaining gap:

The boundary hold is conservative. It is not a source-validated answer
for every masonry or lined-massive hybrid, and it must not become a
silent exact lane.

Fix strategy:

- Keep family-boundary hold behavior for ambiguous masonry/lining
  hybrids.
- Require substrate mass, lining cavity, furring/channel coupling,
  board identity, insulation identity, metric policy, and tolerance
  owner before importing exact masonry or lined-massive rows.
- Keep one-side-lining and attached-board negative boundaries separate
  from independent-leaf triple-leaf behavior.

Tests to add or keep:

- Keep boundary-scan tests that prove no silent large jump occurs at

- Add source-specific negative boundaries whenever a new masonry or
  lined-massive source row is mapped.

## R7 - Material Alias / Coalescing

Symptom:

The calculator can overstate certainty if it globally merges or aliases
materials such as rockwool vs glass-fiber batt, generic gypsum board vs
Type C board, MLV membrane vs limp-mass model, gypsum plaster vs board
finish, or split same-material layers in different roles.

Why it can happen:

Material ids in the local catalog are often generic. Acoustic behavior
depends on role, surface mass, density, flow resistivity, stiffness,
mounting, position, and source tolerance. A correct equivalence in one
role can be wrong in another role.

Current guard evidence:

- Gate G4 local material/topology mapping blocks generic local
  materials from exact NRC-like runtime
- Gate L keeps NRC Type C board / glass-fiber / 92.1 mm cavity /
  support references as adjacent evidence only
- Gate M keeps Type C, support topology, MLV, and plaster follow-on
  blockers open
- `SLICE_FLOOR_LAYER_ORDER_INVARIANCE_PLAN.md` / related gates keep
  global same-material coalescing disallowed when position matters

Remaining gap:

Local equivalence tables for rockwool, Type C gypsum, MLV, and gypsum
plaster do not yet have source/tolerance owners for triple-leaf exact
runtime.

Fix strategy:

- Use per-role material mapping, not global aliasing.
- Require density, flow resistivity or absorber class, surface mass,
  thickness tolerance, and source family owner for equivalence.
- Preserve same-material splits when the split crosses different
  leaves, cavities, or finishes.

Tests to add or keep:

- Add alias negative-boundary tests for rockwool/glass-fiber and
  generic gypsum/Type C before Gate O+ promotion.
- Add local-product positive tests only after source/tolerance owners
  exist.

## R9 - Curve Digitization / Provenance
