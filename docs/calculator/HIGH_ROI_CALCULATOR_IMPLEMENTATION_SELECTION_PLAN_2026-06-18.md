## North Star Lock

DynEcho is a calculator. The selected next owner must improve one or
more of:

- calculable user-entered layer combinations;
- numeric accuracy or bounded error for an owned formula route;
- formula, spectrum, exact-row, compatible-anchor, or building-adapter
  ownership;
- route-required physical input capture;
- metric and basis integrity;
- an explicit `needs_input` or `unsupported` boundary that prevents a
  common wrong answer.

Answer order remains:

1. exact measured/source row only when construction and metric basis
   truly match;
2. same-family, same-basis compatible anchor or bounded delta only when
   the construction boundary is proven;
3. otherwise owned physics/formula route with route-required inputs;
4. otherwise precise `needs_input` or `unsupported`.

## Readiness Audit Of The Six ROI Streams

### 1. User-Material Physical Input Coverage

Strategic ROI: highest. This is where arbitrary user-entered layer
combinations become calculable without searching for exact source rows.

Implementation readiness: high only after narrowing to one route family
and naming all route-required physical inputs. The current calculator
already has useful lanes around custom double-leaf/framed walls,
porous absorber inputs, and low-density floor dynamic-stiffness/load
basis. The next rerank should test whether adjacent user-material
widening can move real request shapes without defaulting missing
topology.

Ready when:

- required inputs are explicit, such as leaf grouping, support topology,
  bridge class, support spacing, cavity depth, absorber flow
  resistivity, dynamic stiffness, load basis, base structure, and layer
  role;
- material names do not affect results when calculation-relevant
  numeric properties, roles, topology, and context are identical;
- missing route-required fields stay `needs_input`;
- the owner can plausibly move at least `newCalculableRequestShapes >=
  2` or `newCalculableTargetOutputs >= 5`.

Not ready when the plan relies on inferred topology, generic absorber
defaults, board leaves treated as massive substrates by mass alone, or
name-based material shortcuts.

### 2. Building Prediction And Flanking Runtime

Strategic ROI: very high. Market-grade outputs need `R'w`, `Dn,w`,
`DnT,w`, `L'n,w`, and `L'nT,w` for real room and junction contexts, not
lab-value copies.

Implementation readiness: medium. Field/building adapter posture exists
in the current chains, but broader flanking needs explicit room and
junction context. This stream is implementation-ready only for a narrow
same-basis direct-plus-flanking owner, such as opening/leak or common
wall residual work, when local formula or compatible-anchor proof exists.

Ready when:

- direct curve or source basis is owned;
- receiving-room volume, separating element area, normalization basis,
  flanking topology, junction class, coupling length, and output basis
  are explicit or already owned by the route;
- lab, field, and building bases stay separate;
- missing context returns `needs_input`.

Not ready when the work copies lab `Rw` into building `R'w`/`DnT,w`, or
hides missing flanking/junction assumptions behind a confidence label.

### 3. Frequency-Band Backbone

Strategic ROI: highest long-term accuracy ceiling. A frequency-first
backbone lets `Rw`, `STC`, `OITC`, `C`, `Ctr`, `Ln,w`, and IIC derive
from an owned one-third-octave curve instead of scalar patches.

Implementation readiness: low to medium for the immediate next slice.
It is ready only when a narrow route already owns, or can safely return,
the frequency curve without speculative retuning. It should not be
selected as a broad architecture project in this rerank.

Ready when:

- one concrete route can return an owned one-third-octave airborne or
  impact curve;
- rating adapters for the requested companions are explicit;
- lab, field, and building adapters are separately owned before reuse;
- negative tests prove unsupported aliases remain blocked.

Not ready when the work synthesizes a curve only to unlock labels, or
requires a global rewrite before moving a bounded request shape.

### 4. Companion Metric Completeness

Strategic ROI: high practical ROI. If a route already owns the spectrum
or rating data, hidden companions such as `C`, `Ctr`, `STC`, or `OITC`
look like calculator gaps to the user.

Implementation readiness: high for owned-spectrum routes and low for
scalar-only routes. This is a good fallback owner when the rerank finds
that a route already carries enough data to expose companions without
metric aliasing.

Ready when:

- each candidate route is classified as owned-spectrum, scalar-only,
  measured-single-number, anchor-delta, or adapter-only;
- derivable companions are exposed only from owned spectrum/rating
  bases;
- scalar-only or measured-single-number lanes keep explicit
  `unsupported` boundaries for missing companions.

Not ready when `C`/`Ctr` are inferred from `Rw` alone, or lab companions
are carried into field/building outputs without an owned adapter.

### 5. Calibration And Holdout Packets

Strategic ROI: high for accuracy, especially for market-above formula
quality. Measured evidence should tighten or validate a family, not
replace the calculator with a row catalog.

Implementation readiness: medium to low unless a rights-safe local
holdout packet already exists for one family. This stream may require
internet research, but only to find standards, manufacturer technical
data, papers, or official documentation with same-family and same-basis
proof.

Ready when:

- one family is selected;
- source rights and metric basis are clear;
- rows are assigned as exact override, compatible anchor, calibration,
  or holdout;
- formula retuning or budget tightening has an explicit before/after
  error test.

Not ready when it becomes broad source crawling, near-miss row reuse, or
source-absent validation of formula outputs.

### 6. Opening/Leak, Common-Wall, Open-Box/Open-Web Residuals

Strategic ROI: high and concrete because these are named current
residuals likely to unlock real user constructions.

Implementation readiness: medium. They are good rerank candidates, but
only if they move actual target outputs through owned formulas,
same-basis exact rows, or compatible anchors.

Ready when:

- wall opening/leak or common-wall work has same-basis proof for the
  affected airborne or building outputs;
- open-box/open-web floor work can move `Ln,w`, `DeltaLw`, `L'n,w`,
  `L'nT,w`, `IIC`, or `AIIC`;
- required geometry, area ratio, seal/leak path, joist/open-web
  topology, resilient layer role, dynamic stiffness, and load basis are
  explicit as needed;
- adjacent unsupported cases remain unsupported.

Not ready when the route has no owned formula/anchor, when missing
physical inputs are defaulted, or when a finite source row is applied to
nearby but unproven constructions.

## Iteration 3 - Implementation-Ready Gates

Each candidate must pass these gates before becoming the selected
runtime owner.

Gate 1 - route basis:
the route must name its formula, spectrum owner, exact measured row, or
compatible anchor. If none exists, it cannot be a runtime owner.

Gate 2 - physical inputs:
all route-required physical inputs must be explicit in request,
material, layer, topology, or context data. Missing values must return
`needs_input`, not defaults.

Gate 3 - target movement:
prefer candidates with `newCalculableRequestShapes >= 2`,
`newCalculableTargetOutputs >= 5`, `runtimeBasisPromotions >= 2`, or a
reproduced wrong-basis/wrong-answer fix for a common construction.

Gate 4 - metric and basis integrity:
no `Rw`/`STC`/`C`/`Ctr` aliasing, no `IIC`/`AIIC` aliasing, and no
lab-to-field/building copy without an owned adapter.

Gate 5 - bounded blast radius:
the owner must be testable with positive, target-output independence,
missing-input, unsupported-adjacent, and non-goal tests. If the slice
needs a broad rewrite, split it.

Gate 6 - evidence confidence:
local evidence is preferred. Use internet research only when needed to
prove a formula, standard, measured-anchor basis, or manufacturer data
for the selected narrow family.

## Iteration 4 - Selection Rule For The Next Owner

The active rerank should select one owner only after applying the gates.
The preferred order is:

1. select user-material physical input widening if local probes prove
   complete physical inputs unlock multiple real request shapes;
2. otherwise select building/flanking, opening/leak, common-wall, or
   open-box/open-web residual work if same-basis formula/anchor proof
   exists and actual field/building/impact outputs move;
3. otherwise select companion metric completeness if a route already
   owns spectrum/rating data and hidden companions can be exposed
   without aliasing;
4. otherwise select a narrow frequency-band backbone route only if it is
   implementable without a global rewrite;
5. otherwise create a calibration/holdout or formula-evidence research
   plan and do not move runtime.

The selected owner plan must name route family, target outputs,
required inputs, formula or anchor basis, unsupported boundaries,
estimated counters, and validation commands before runtime code changes
begin.

- the route depends on guessed topology or defaulted physical inputs;
- metric support comes from scalar aliasing;
- lab, field, and building bases are conflated;
- ASTM and ISO companions are substituted for one another;
- a source row is only a near match and no compatible-anchor rule is
  owned;
- the work is UI/report/display/confidence prose without calculator
  movement;
- the implementation would become a broad source crawl or unbounded
  architecture rewrite.
