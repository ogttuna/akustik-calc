## Candidate Families To Compare

The rerank must compare calculator-first candidates only:

- **Project/user measured anchor bridge expansion prerequisites**:
  measured `STC`, `C`, `Ctr`, or frequency-band anchors are eligible
  only if their own schema, metric basis, and exact fingerprint are
  owned first. Do not alias from the landed single-number `Rw` anchor.
- **User-material physical input coverage**: route-required inputs for
  arbitrary custom wall/floor stacks, especially fields that unlock
  owned formula routes rather than source-row lookups.
- **Building prediction + flanking runtime**: bounded `R'w`, `Dn,w`,
  `DnT,w`, `L'n,w`, and `L'nT,w` owners where lab values and building
  context are both physically owned.
- **Frequency-band backbone prerequisites**: schema/route work that can
  derive `Rw`, `STC`, `C`, `Ctr`, `OITC`, `Ln,w`, or `IIC` from owned
  bands without scalar aliasing.
- **Companion metric completeness**: only routes where the same owned
  spectrum or direct formula already carries companion values.
- **Calibration / holdout packets**: small, same-family, same-basis,
  rights-safe packets; no broad source crawl.

## No-Go Conditions

Reject candidates that primarily do any of the following:

- crawl broad sources without a bounded same-basis target;
- polish UI or report assistant surfaces without opening or protecting
  calculator behavior;
- retune formulas without an accepted holdout or owned evidence packet;
- publish companion metrics from scalar `Rw` anchors;
- copy lab values into field/building outputs without context and
  flanking ownership;
- change runtime before the rerank contract lands.
