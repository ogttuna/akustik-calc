Move DynEcho from narrow target-output owner chains toward bounded
route-family campaigns that make more user-entered assemblies calculate

## Selection Rule

The next post-refresh candidate should be selected only if it can answer
all of these questions concretely:

1. Which route family is being improved?
2. Which target outputs will become calculable or more accurately owned?
3. Which physical inputs does the route require?
4. Which formula, spectrum owner, anchor, or adapter owns the value?
5. Which adjacent metrics, bases, or families must stay unsupported or
   `needs_input`?
6. What are the estimated runtime movements?

Preferred minimum movement for a runtime owner:

- `newCalculableRequestShapes >= 2`, or
- `newCalculableTargetOutputs >= 5`, or
- `runtimeBasisPromotions >= 2`, or
- a meaningful accuracy promotion for an already common route.

No-runtime work is allowed only when it selects or protects a concrete
next runtime/accuracy owner. Do not allow two consecutive no-runtime
planning slices unless the second one is explicitly required to prevent
wrong calculator answers.
