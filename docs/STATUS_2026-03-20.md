# DynEcho Status Snapshot — 2026-03-20

This note captures the current implementation reality of DynEcho against the product goal:

- professional acoustic calculator
- web-first delivery
- portable engine
- explicit evidence posture
- consultant-grade UI/UX

Use this document as the short operational companion to `docs/PROJECT_PLAN.md`.

## 1. Product Goal Check

DynEcho is intended to calculate and present acoustic outputs such as:

- `Rw`, `R'w`
- `Dn,w`, `Dn,A`, `DnT,w`, `DnT,A`, `DnT,A,k`
- `Ln,w`, `Ln,w+CI`, `L'n,w`, `L'nT,w`, `LnT,A`
- `DeltaLw`

The product standard is not just "returns a number". The standard is:

- mathematically reasonable within defended corridors
- explicit about evidence class
- clear about unsupported or staged outputs
- usable in a professional workflow without hidden solver posture

## 2. Current Reality

### 2.1 Engine

The engine is now in a materially better state than the earlier migration shell.

What is live today:

- calibrated airborne outputs: `Rw`, `STC`, `C`, `Ctr`
- field airborne continuation: `R'w`, `Dn,w`, `Dn,A`, `DnT,w`, `DnT,A`, `DnT,A,k`
- exact impact import lanes
- exact floor-family impact rows
- official impact product rows
- heavy-floor scoped predictor lane
- labeled low-confidence published-family fallback lanes for the remaining floor corridors

What was repaired in the current pass:

- impact-only upstream parity is green again
- assembly impact parity is green again
- low-confidence floor-family fallback behavior is aligned with upstream instead of drifting locally
- timber bare-floor low-confidence lanes now expose finite airborne companions when they genuinely exist

Current explicit non-goals / unsupported lanes:

- `IIC`
- `AIIC`
- `NISR`
- `ISR`
- `LIIC`
- `LIR`
- `HIIC`

Those outputs are intentionally kept explicit instead of fabricated.

### 2.2 Web Product

The product now has two faces:

- guided shell: closer to the real product surface
- advanced desk: expert/operator surface

The guided shell is currently the stronger product face because it:

- explains corridor posture more clearly
- keeps missing topology visible
- reduces operator overload

The advanced desk is powerful but still dense. It is now more stable than before:

- `/workbench` standalone start path no longer returns `500`
- chart panels are guarded against zero-size startup rendering
- production browser validation now runs without chart warnings

### 2.3 Delivery Posture

The app is no longer only a calculator shell. It now has working product-level surfaces for:

- requested-output tracking
- capability mapping
- field-risk framing
- scenario comparison
- report/export composition

But this is still not full consultant-grade delivery. Reporting exists, but trust/provenance still needs one more tightening pass.

## 3. What Is Still Missing

### 3.1 Math / Solver Gaps

The biggest remaining gaps are not random bugs. They are scope gaps:

- ASTM impact families remain staged
- field continuation needs deeper benchmark coverage and stronger parity discipline
- some guide-driven outputs are still honest placeholders rather than full standards-backed solvers

### 3.2 Product Gaps

The product still needs:

- stronger consultant-facing provenance surfaces
- clearer distinction between exact, estimate, bound, low-confidence, and guided outputs
- less density in the advanced desk
- a more deliberate primary workflow around the guided shell

## 4. Recommended Execution Order

### Phase 1 — Field Output Hardening

Priority:

- `R'w`
- `Dn,w`
- `Dn,A`
- `DnT,w`
- `DnT,A`
- `DnT,A,k`

Exit criteria:

- benchmark-backed corridors
- explicit parity acceptance where upstream comparison is meaningful
- clean target-output posture on both engine and UI surfaces

### Phase 2 — Guided Shell As Primary Product

Priority:

- treat the guided shell as the default user journey
- keep advanced desk as expert/operator mode
- reduce lane confusion and panel density

Exit criteria:

- first-time professional user can understand output posture without opening diagnostics
- advanced desk becomes optional depth, not required interpretation

### Phase 3 — Provenance And Reporting

Priority:

- source references
- confidence posture
- active corridor
- assumptions
- export narrative

Exit criteria:

- consultant can lift an output into a brief or memo without rewriting the solver posture by hand

### Phase 4 — ASTM / Extended Impact Families

Priority:

- `IIC`
- `AIIC`
- `NISR`
- `ISR`
- `LIIC`
- `LIR`
- `HIIC`

Exit criteria:

- standards-backed adapters, not marketing placeholders

## 5. Product Judgement

DynEcho is now beyond the "migration demo" stage.

It is not yet a universal consultant-grade acoustic calculator, but it is already a serious web product foundation with:

- a defended engine boundary
- meaningful parity discipline
- explicit evidence posture
- a credible guided workflow

The next highest-value move is not another broad UI sweep. The next highest-value move is field-output hardening plus provenance/report tightening.
